Accessing secrets in Rails application

Sometimes you need to keep certain things private (tokens, passwords, etc.). Rails comes with a handy secrets.yml file in which you can store secrets. However, this is now deprecated in favor of using environmental variables. If you do choose to still store secrets in secrets.yml, you should remember to uncomment config/secrets.yml from .gitignore so that they don’t get pushed to revision history.


In case you need to remove the file from most recent commit:

    $ git rm –cached filename
    $ git commit –amend -CHEAD
    $ git push

Storing environmental variables in linux is very straight forward:

    $ export NAME_OF_VARIABLE=value

To make sure it was saved correctly:

    $ printenv NAME_OF_VARIABLE

So now that you have some secrets, how do you access them? Well that depends on where you stored them. If they are in secrets.yml, then, on initialization, they are loaded into the following object: Rails.application.secrets and can be accessed just like any other object, such as Rails.application.secrets.secret_key_base. If you went with environmental variables, its as easy as ENV[‘NAME_OF_VARIABLE’].

These are both Ruby commands, so if you would like to use them in the asset pipeline, you should use .js.erb or html.erb files and use properly formated erb (<%= %>). If you are using them this way, just remember that you are bringing secrets to the client side so they could be exposed to users. If you must keep them private, it would be best to keep them server side and use Ruby objects.
